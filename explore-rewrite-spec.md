# Explore Page Rewrite — Model Playground Spec

A targeted refactor of `streamlit_app/pages/1_📊_Explore.py`. The current page shows raw DOT data with no connection to the trained model. This spec rewrites all four tabs as **model behavior explorers** — each view asks "what does the model think, and how does that compare to reality?"

---

## Context to read first

Before writing any code, read these files in the repo to understand the shape of what you're working with:

1. **`streamlit_app/pages/1_📊_Explore.py`** — the file you're replacing. Understand the current tab structure so you can preserve what works (layout, tab names where appropriate, altair styling patterns).
2. **`streamlit_app/components/prediction.py`** — contains `predict_fare()`, `get_route_history()`, `get_top_routes()`, `carrier_tier_fn()`, `lookup_both_directions()`. You'll use and extend these; **do not duplicate logic**.
3. **`streamlit_app/components/artifacts.py`** — contains `load_artifacts()`, `load_route_history_csv()`, `CARRIER_NAMES`, `QUARTER_LABELS`, `MODEL_METRICS`. The artifact dict has `model`, `feature_cols`, encoders, and all lookups needed to reconstruct feature rows.
4. **`streamlit_app/streamlit_app.py`** — the Estimator landing page. Match its visual language (hero metric cards, muted editorial tone, altair charts with `#0c4a6e` as primary color, `#dc2626` for "selected/highlighted" state).

The artifact pickle is at `streamlit_app/data/us_flight_fare_artifacts.pkl` and `route_history.csv` is at `streamlit_app/data/route_history.csv`. Both are generated by the notebook.

---

## Design principle

**Every chart must show "actual vs model" or "what the model learned."** If a chart only shows raw data without any model output, it doesn't belong on this page — that's what the notebook EDA is for. The Explore page exists to let users probe the model's behavior on real examples.

Color conventions (keep consistent across all tabs):
- **Actual / observed** — `#111827` (near-black), solid line, slightly thicker
- **Predicted / model** — `#dc2626` (red), dashed line
- **Residual positive (model under-predicts)** — `#dc2626` (red)
- **Residual negative (model over-predicts)** — `#2563eb` (blue)
- **Selected/highlighted point** — `#f59e0b` (amber), larger marker

---

## Performance requirements

The Explore page will batch-predict on thousands of rows. Wrap every expensive function in `@st.cache_data` keyed on a hash of the inputs (use `_art` leading-underscore arg convention since artifacts aren't hashable). Target budgets:

- Any user interaction (tab switch, dropdown change) → UI updates within 1.5 seconds
- Initial tab load on cold cache → within 4 seconds
- If batch prediction would exceed that, sample the data (e.g., take the top 100 routes, not all 1000)

Show a `st.spinner("Computing model predictions...")` block around any slow operation so users see that something is happening.

## New helpers to add to `components/prediction.py`

Add these functions — they're reused across tabs.

### `batch_predict(art, df, feature_cols=None) -> np.ndarray`

Takes a DataFrame with the raw columns needed to build feature rows, returns an array of predictions. Reuses the same feature-engineering logic as `predict_fare()` but vectorized. Add a `@st.cache_data` decorator.

Implementation sketch:

```python
@st.cache_data
def batch_predict(_art, df_hash_key: str, df: pd.DataFrame) -> np.ndarray:
    """Predict fares for a DataFrame of (city1, city2, carrier, Year, quarter) rows.
    
    df_hash_key is a string used for caching (pandas DataFrames aren't hashable).
    Caller must pass a stable string like "top_routes_2024" or a hash of the df.
    """
    # Build feature rows vectorially
    rows = df.copy()
    rows['log_distance']    = np.log1p(rows['nsmiles'])
    rows['log_passengers']  = np.log1p(rows['passengers'])
    rows['is_long_haul']    = (rows['nsmiles'] > 1500).astype(int)
    rows['distance_bucket'] = pd.cut(rows['nsmiles'], bins=[0, 500, 1000, 1500, np.inf],
                                     labels=[0, 1, 2, 3]).astype(int)
    rows['quarter_sin'] = np.sin(2 * np.pi * rows['quarter'] / 4)
    rows['quarter_cos'] = np.cos(2 * np.pi * rows['quarter'] / 4)
    rows['is_covid_era'] = rows['Year'].isin([2020, 2021]).astype(int)
    rows['carrier_tier'] = rows['carrier_lg'].apply(
        lambda c: carrier_tier_fn(c, _art))
    rows['carrier_lg_enc'] = _art['carrier_encoder'].transform(rows['carrier_lg'])
    rows['city1_freq'] = rows['city1'].map(_art['city1_freq']).fillna(0.001)
    rows['city2_freq'] = rows['city2'].map(_art['city2_freq']).fillna(0.001)
    rows['route_popularity'] = rows.apply(
        lambda r: _art['route_pop_map'].get((r['city1'], r['city2']),
                    _art['route_pop_map'].get((r['city2'], r['city1']),
                    _art['global_mean_pop'])), axis=1)
    
    # Macro features per (Year, quarter)
    for col in ['jet_fuel_usd_per_gal', 'jet_fuel_lag_1q', 'jet_fuel_yoy_change',
                'cpi', 'cpi_yoy_change']:
        rows[col] = rows.apply(
            lambda r: _art['macro_lookup'].get((int(r['Year']), int(r['quarter'])),
                                                _art['macro_defaults']).get(col), axis=1)
    
    X = rows[_art['feature_cols']]
    return _art['model'].predict(X)
```

### `get_route_with_predictions(art, city1, city2, carrier=None) -> pd.DataFrame`

Returns the historical fare data for a route **plus a `predicted` column** from the model. If `carrier` is None, picks the most common carrier on that route (mode of `carrier_lg`). This is the workhorse function for the Compare Routes and Seasonality tabs.

### `get_top_routes_with_predictions(art, limit=100) -> pd.DataFrame`

Extends `get_top_routes()` by adding three columns: `predicted_fare`, `actual_fare`, `residual`. The `actual_fare` is the mean fare for that route across all historical observations; the `predicted_fare` uses the most recent year (2025 Q2 as a sensible default) and the route's dominant carrier.

### `get_feature_contributions(art, city1, city2, carrier, year, quarter) -> list[dict]`

Returns a ranked list of the top feature contributions to a specific prediction. Use a simple counterfactual approach: for each of 8-10 "big" features (nsmiles, carrier_tier, large_ms, Year, quarter_sin, jet_fuel_usd_per_gal, cpi, route_popularity), compute `prediction_with_feature - prediction_with_feature_at_dataset_mean`. Return the top 5 by absolute contribution.

**Don't use SHAP** — too heavy a dependency. This lightweight counterfactual approach is good enough for a dashboard.

## Tab 1 — Route Residuals (replaces "Top Routes")

**Goal:** show which routes the model handles well and which it struggles on.

### Layout

- Header: "Where does the model agree with reality?"
- Subtext: "Top routes by passenger volume. Bar height = mean actual fare. Color = model residual (how far off the model's average prediction is from observed reality). Red means the model under-predicts this route."
- Slider: "Show top N routes" (default 30, range 10-100)
- Sort toggle: radio with options `["By passenger volume", "By |residual| (hardest first)", "By |residual| (easiest first)"]`

### Chart

Horizontal bar chart, one row per route:
- Y-axis: route name
- X-axis: mean actual fare
- Color: residual (diverging `redblue` scale centered on 0; positive residuals red, negative blue)
- Tooltip: route, distance, passengers, actual fare mean, predicted fare mean, residual, % error
- Height: `max(400, 22 * len(df))` for readability

### Below the chart

Two metric cards:
- "Routes where the model excels" → count of routes with |residual| < $10
- "Routes where the model struggles" → count of routes with |residual| > $30, plus the top 3 worst routes named explicitly

### Data flow

Call `get_top_routes_with_predictions()`, sort by the user's chosen criterion, slice to N. Chart it.

## Tab 2 — Actual vs Predicted Route History (replaces "Compare Routes")

**Goal:** let users see how well the model tracks reality on specific routes over time, and compare model accuracy across two routes side-by-side.

### Layout

Two route pickers side by side (same as current). A third dropdown: "Carrier (both routes)" — if None, each route uses its dominant carrier.

### Chart

Faceted Altair chart: two sub-plots stacked, one per route. Each shows:
- Black solid line: actual fare over time
- Red dashed line: model prediction over time (same carrier, same year-quarter points as actual observations)
- Hover tooltip: year, quarter, actual, predicted, residual

Add a small MAE badge above each subplot: `Route A — MAE on this route: $X.XX`.

### Below the charts

A comparison table with 5 rows:
- Mean actual fare
- Mean predicted fare
- MAE
- Best year (smallest |residual|)
- Worst year (largest |residual|)

Show Route A and Route B as two columns. Minor win indicator in the last column (📈 lower MAE wins accuracy contest).

### Failure case copy

If either route has fewer than 8 historical observations, show an `st.info()` saying "Route has limited history; model accuracy estimates may be unreliable."

## Tab 3 — Carrier Response Curves (replaces "Carrier Profile")

**Goal:** show what the model learned about each carrier by holding all other features fixed and varying one input at a time.

### Layout

- Carrier dropdown (same as current, with tier badge)
- Below the badge: brief text explaining what "response curve" means, one sentence max.
- Three tabs (nested `st.tabs` inside this tab):
  - **vs Distance** — predicted fare as distance varies from 100 to 3000 mi, holding year=2024, quarter=2, large_ms=0.5 fixed. All other features at their dataset median.
  - **vs Year** — predicted fare as year varies 2010-2025, holding distance=1000, quarter=2, large_ms=0.5.
  - **vs Market Share** — predicted fare as `large_ms` varies from 0.1 to 1.0, holding distance=1000, year=2024, quarter=2.

### For each nested tab

A line chart showing the predicted fare curve for this carrier (red) overlaid on the equivalent curve for a baseline reference carrier (gray — use whichever legacy carrier is most common in training data, likely "AA"). This lets the user see how *this* carrier differs from a reference.

Tooltip: x-value, predicted fare for this carrier, predicted fare for reference, difference.

Below each chart, one bold-text takeaway auto-computed from the curve:
- "Model predicts that **{carrier}** charges ${X} more than {reference_carrier} at 1500-mile distance."
- If the difference is < $5, phrase it as "Model predicts that **{carrier}** prices similarly to {reference_carrier}."

### Implementation note

Building the response curve is a batch prediction over a synthetic DataFrame. Use `batch_predict()` with a cache key like `f"response_{carrier}_{varying_feature}"`. Each curve is 50 points, so 50 model calls in a single vectorized prediction — should be fast.

## Tab 4 — Seasonality: Actual vs Learned (replaces "Seasonality")

**Goal:** show whether the model successfully captured the seasonal patterns in the data.

### Layout

- Two route pickers (same style as current)
- Below the pickers: two heatmaps **side by side** with shared color scale:
  - **Left:** actual fare, year × quarter, from `get_route_history()`
  - **Right:** predicted fare, year × quarter, same route and the dominant carrier on that route
- Third heatmap below, **full width:** residual (actual - predicted), year × quarter. Diverging color scale centered on 0.

Overlay fare values as text on each cell (white on dark, black on light, same treatment as current seasonality tab).

### Below the heatmaps

Auto-generated insight paragraph using the actual data, e.g.:
- "The **actual** data shows Q2 and Q3 are consistently priciest (summer vacation peak)."
- "The **model** captures this pattern but slightly under-predicts Q3 spikes in 2023-2025 — a limitation we discuss in §10.5."
- These insights should be computed from the data, not hardcoded. Logic: find which quarter has the highest actual mean and highest predicted mean; if they match, say so; if they differ, call it out.

### Failure case

If route has fewer than 8 observations, show an info banner (same as Tab 2).

## Error handling

Every batch prediction path must handle:

- **Missing route_history.csv** — if `load_route_history_csv()` returns an empty DataFrame, show `st.warning()` at the top of any tab that needs it, with clear instructions to generate the file from notebook §12.1.
- **Unknown carrier/city** — if a user-selected carrier isn't in `art['carriers']`, show an error message and don't crash.
- **Empty historical data for a route** — show the info banner referenced in Tab 2 and Tab 4.
- **Model prediction failure** — wrap batch calls in try/except and show a friendly message rather than a traceback.

## Acceptance criteria

Before you declare the rewrite done, verify:

1. **Every tab calls the model at least once.** Grep for `model.predict` or `batch_predict` in each tab's code path; should find matches in all four tabs.
2. **No tab shows only raw DOT data** without at least one model-derived visualization or metric.
3. **All four tabs run in under 4 seconds on a cold cache, under 1.5 seconds after warmup.** Test with Python's `time.perf_counter()` if unsure.
4. **The page degrades gracefully without `route_history.csv`** — it shows informative warnings instead of crashing.
5. **Color conventions are consistent** — actual is near-black, predicted is red-dashed, residuals are diverging red/blue, selected is amber.
6. **No code duplication** — any logic used in more than one tab is factored into `components/prediction.py`.
7. **The "open in Estimator" hook works** on at least one tab (Tab 1, Tab 2, or Tab 4).

## What not to do

- **Do not add new Python dependencies.** Stick to Streamlit + pandas + numpy + altair + scikit-learn, already in `requirements.txt`. No SHAP, no Plotly, no Folium, no Seaborn.
- **Do not re-invent functions already in `components/prediction.py`.** Extend that module; don't copy-paste logic.
- **Do not change the Estimator or About pages.** Scope is strictly the Explore page rewrite and the new helpers in `prediction.py`.
- **Do not add a map.** The current setup doesn't have airport coordinates and adding them is out of scope for this rewrite.
- **Do not increase pickle size.** If you need new lookups, compute them on the fly using existing artifact data.

## Suggested order of implementation

1. First, add the four new helpers to `components/prediction.py` and verify they work by printing sample outputs in a scratch cell or REPL.
2. Then build Tab 1 (Route Residuals) — the simplest tab that needs batch prediction. Getting this right validates the caching and performance approach before building the more complex tabs.
3. Then Tab 4 (Seasonality) — reuses `get_route_with_predictions()` which is simple to build.
4. Then Tab 2 (Actual vs Predicted) — same helper plus the MAE metric.
5. Last, Tab 3 (Carrier Response Curves) — most complex because it requires synthetic feature rows.

Each tab should work end-to-end on its own before moving to the next. That way, if you run out of time, you ship 3 working tabs instead of 4 half-built ones.
